# 2026-04-06 多角色系统与语音陪读功能完整记录

> 本文档记录了从多角色（Persona）系统设计到 ElevenLabs 语音陪读集成的完整开发过程，包括每一个设计决策的前因后果、遇到的实际困难和最终采用的方案。

## 一、多角色系统

### 1.1 需求起源

用户发现林亦的原项目 mad-professor-public 视频中有切换角色的功能（雷电将军、可莉），但原项目没有 UI 切换入口，只能手动改代码。用户提出将"多角色"作为核心功能，每次对话选择不同角色，不同角色对应不同的系统提示词和语音音色。

### 1.2 架构设计评审

在动手之前，进行了全面的架构评审：

**决策 1：Persona 绑定在哪一层？**
- 最初决定 per-thread（角色与对话绑定），创建时确定
- 后来用户要求可以在**当前对话中直接切换角色**，不开新对话
- 最终方案：角色是动态的，通过 `x-persona` header 传递，每次请求可以不同。同时 PATCH 端点持久化到 DB

**决策 2：Persona 定义放在哪里？**
- 最初所有 prompt 堆在 model.ts 的 SYSTEM_PROMPT 常量中
- 第一次重构：拆分为 `TOOL_INSTRUCTIONS`（通用工具说明）+ 各角色 prompt
- 第二次重构：每个角色独立文件 `src/worker/personas/*.ts`
- 最终评审发现显示数据（emoji、名称、颜色、placeholder）在 Chat.tsx、VoiceThread.tsx、personas/*.ts 三处重复
- 第三次重构：`Persona` 接口统一包含所有显示字段（emoji, title, desc, accentColor, gradient, border, glow, placeholder），客户端直接 import `PERSONAS`，删除所有重复定义

**决策 3：角色选择 UI**
- 替换了原有的"AI 食谱助手"开屏页（ThreadWelcome）
- 设计为游戏化角色选择卡片（gradient 背景、emoji 头像、选中发光效果）
- 暴躁教授为默认选中

## 二、ElevenLabs 语音集成

### 2.1 STT + TTS 管道模式（之前已有）

项目已有的语音功能：
- **STT**：ElevenLabs Scribe v2 Realtime（DictationAdapter）
- **TTS**：ElevenLabs eleven_v3（SpeechSynthesisAdapter）
- 用户按住麦克风说话→文字→发送→AI 回复→朗读
- opencc-js 处理繁体→简体中文转换

### 2.2 自动朗读（Auto-TTS）

- 添加 AutoTTSToggle 按钮（Volume2/VolumeOff 图标），默认开启
- 设置持久化到 localStorage（`settings:autoTTS`）
- AutoSpeakWatcher 监听 `isRunning` true→false 转换
- **关键修复**：最初直接调用 `ttsAdapter.speak()`，消息 UI 不显示朗读状态。改为 `aui.thread().message({ id }).speak()` 走 runtime 通道
- 用户发新消息时自动停止当前朗读（`stopSpeaking()`）
- 语音模式激活时跳过自动朗读（避免双重音频）

### 2.3 每角色语音参数统一

- `Persona` 接口增加 `voiceSpeed` 和 `voiceStability`
- TTS adapter 增加 `voiceSpeed`/`voiceStability` 字段
- 后端 `/api/tts` 传入 `voice_settings`（speed + stability + similarity_boost）
- 两种模式（文本朗读和语音陪读）共享同一套角色音色配置

## 三、ElevenLabs Conversational AI（语音陪读模式）

### 3.1 设计构想

用户的核心想法：在文档查看器中打开一篇文档，点击"语音陪读"按钮，右侧面板切换为实时语音对话模式。文档全文注入系统提示词，使用当前角色的人设和音色。

### 3.2 可行性评估

**ElevenLabs Conversational AI 的能力**：
- `VoiceConversation.startSession()` 支持 `overrides` 参数
- 可以动态替换：system prompt、firstMessage、voiceId、speed、stability、language
- 需要在 Dashboard 的 Agent Security 中开启所有 override 开关
- 使用 signed URL 认证（通过后端代理）

**与现有架构的关系**：
- 语音模式**不使用工具调用**（纯对话）
- 语音模式的 LLM 是 ElevenLabs Agent 内部的（非我们的 OpenRouter 后端）
- 两个 LLM 完全独立——text 模式用 OpenRouter，voice 模式用 ElevenLabs

### 3.3 架构演进（三次重大重构）

**第一版：通过 assistant-ui RealtimeVoiceAdapter**
- 创建 `ElevenLabsVoiceAdapter` 类实现 `RealtimeVoiceAdapter` 接口
- 注册在 `useAISDKRuntime` 的 `adapters.voice`
- **严重问题**：`emitTranscript` 在 assistant-ui runtime 中创建线程消息 → 触发 `initialize()` → 创建独立线程 → 退出后 text 线程状态损坏 → 永远 loading

**第二版：standalone 但 adapters 对象不稳定**
- 从 runtime adapters 中移除 voice adapter
- VoiceThread 自己管理 WebSocket 连接和消息状态
- **问题**：最初将 voice adapter 有条件地加入 `stableAdapters`（`voiceMode.active ? { voice } : {}`），导致每次切换时 adapters 对象引用变化 → runtime 重新初始化 → loading 卡死

**第三版（最终方案）：完全 standalone**
- voice adapter 从 `stableAdapters` 中彻底移除
- `startVoiceSession()` 改为纯函数（非 class）
- VoiceThread 用自己的 `useState` 管理消息、状态、音量
- useEffect 在 mount 时自动连接，cleanup 自动断开
- **不创建线程，不影响 text 模式的任何状态**

### 3.4 消息顺序问题

**问题**：ElevenLabs 的 `onMessage` 时序是 AI 回复先到，用户 transcript 后到。导致 AI 的回复显示在用户消息上方。

**多次尝试**：
1. "同角色替换"逻辑 → 开场白被替换消失
2. buffer 所有 AI 消息等 user 到 → React StrictMode 下 ref 被双重消费，AI 消息丢失
3. 全局 Set 去重 → StrictMode 清空时机不对

**最终方案**：
- 开场白直接显示（`msgs.length === 0` 时）
- 后续 AI 消息进 `pendingAI`（存在 **state** 中而非 ref，避免 StrictMode 问题）
- User 消息到达时 flush：`[...msgs, userMsg, pendingAI]`
- 后续轮次（第二轮开始）ElevenLabs 时序恢复正常，直接 append

### 3.5 双向历史合并

**Text → Voice**：
- `enterVoiceMode` 时读取 `runtime.thread.getState().messages`，取最近 10 条，格式化为 `学生：xxx / 导师：xxx` 摘要
- 注入 voice 的 system prompt 尾部

**Voice → Text**：
- VoiceThread 退出时通过 `onExit(messages)` 传回 transcript 数组
- RuntimeProvider 的 `exitVoiceMode` 将消息存入 `pendingVoiceMsgs`
- `useMyRuntime` 的 useEffect 检测 `voiceMode.active` 变为 false 时消费
- 插入分隔符 `🎙 进入语音陪读`（user 角色）+ 所有 voice 消息（AI 消息加 🎙 前缀）
- 使用 `chat.setMessages()` 静默注入（不触发 AI 回复）
- 同时通过 `POST /api/threads/:id/voice-messages` 持久化到 DB

**曾经的错误方案**：
- 最初用 `x-voice-summary` header 传中文摘要 → `Failed to read headers: non ISO-8859-1 code point` → 所有后续请求全部崩溃
- 然后用 `thread.append()` 注入 → 每条消息触发 AI 回复 → 连锁反应，AI 疯狂输出

### 3.6 连接/断开问题

**React StrictMode 双连接**：useEffect 在 StrictMode 下执行两次。修复：cleanup 函数 await sessionPromise 再 disconnect。

**挂断后自动重连**：最初 AutoConnect 监听 "ended" 状态触发 `exitVoiceMode` → 竞态条件导致重新进入。修复：去掉 AutoConnect 的 onDisconnect 回调，改为挂断按钮直接调用 `onExit`。

### 3.7 WebGL VoiceOrb

- 从 elevenlabs-voice 参考项目移植 WebGL Simplex noise 着色器
- 改造为 props 驱动（接收 `state`、`volume`、`accentColor`），不依赖 assistant-ui hooks
- **shader 编译失败问题**：第一次手动压缩 shader 代码导致语法错误。恢复为原版逐字复制后解决
- **运行时崩溃问题**：`makeColors()` 每次 render 创建新数组 → `useCallback` 依赖 `colors` → WebGL 不断重建 → context lost。修复：`useMemo` 缓存 colors
- 根据角色 `accentColor` 自动生成三色调色板

### 3.8 方括号标签过滤

ElevenLabs 的 transcript 包含 `[sigh]`、`[laughs]`、`[背景音乐]` 等标签。在 adapter 的 `onMessage` 中用 `/\[[^\]]*\]/g` 正则过滤。

## 四、工具调用系统修复

### 4.1 ask_user 无限循环

**问题**：AI 调用 ask_user → 用户选择 → `sendAutomaticallyWhen` 续发 → AI 又调用 ask_user → 无限循环

**根因**：`stripUnresolvedToolCalls` 函数扫描 `type === "tool-result"` 来判断哪些 tool call 已解决。但 assistant-ui 的 HITL 工具结果存储在同一个 part 内（`tool-ask_user` 的 `result` 字段），而非独立的 `tool-result` part。函数找不到任何 `tool-result` → 认为所有 tool call 未解决 → 全部删除 → LLM 看不到用户选择 → 再次调用

**修复**：重写 `stripUnresolvedToolCalls`——只删除 `state === "input-available"` 且没有 `result`/`output` 的 part。有 `output`（非 HITL 如 update_recipe）或 `result`（已回答的 HITL）的 part 保留

### 4.2 HITL 结果不持久化

**问题**：用户在 ask_user 上选择后，结果只在前端 runtime 内存中更新，从未回写 DB。刷新后从 DB 加载的还是原始 `input-available` 状态

**修复**：
1. `saveMessages` 从 `onConflictDoNothing` 改为 `onConflictDoUpdate`（upsert `parts` 字段）
2. 后端 `/api/chat` 在处理每个请求时，扫描所有输入的 assistant 消息，找到含已解决 tool 的（`result != null || output != null`），upsert 到 DB

### 4.3 onFinish 累积消息覆盖

**问题**：`sendAutomaticallyWhen` 续发时，`onFinish` 的 `finishedMessages` 包含了被修改/累积后的旧消息。用 upsert 保存后，DB 中的消息被覆盖为包含重复 reasoning parts 的版本

**修复**：收集输入消息的所有 ID（`inputIds`），`onFinish` 只保存不在 `inputIds` 中的真正新消息

### 4.4 思考过程重复显示

**问题**：`sendAutomaticallyWhen` 续发把新一轮的响应追加到同一条 assistant 消息中，导致 reasoning parts 累积（每轮都重复之前所有轮次的思考）

**多次尝试**：
1. 全局 Set 按 `messageId::text` 去重 → React StrictMode 双渲染消费 Set → 所有 reasoning 被隐藏
2. `queueMicrotask` 清空 → 流式 render 间清空太快 → Set 永远为空 → 去重失效
3. `content.find()` 对象引用比较 → assistant-ui 每次 render 创建新对象 → 永远不等 → 去重失效
4. 通过 Context 传递 dupTexts + counter Map → 多种原因仍不生效
5. `performance.now()` 时间窗口 → 不可靠

**最终方案**：最简单的方式——ReasoningPart 组件检查 `hasEverStreamedRef.current`（是否曾经经历过 streaming）和 `duration`（是否有计时）。如果从未 streaming 过且无 duration，说明是从历史加载或续发轮次复制来的 → `return null` 隐藏

### 4.5 doc_search 重命名

将 `doc_search` 改为 `doc_rag_search`，在工具描述和系统提示词中注明 RAG 语义检索的细节（chunk size ~1500 字、128 字重叠、向量相似度匹配），帮助模型理解工具能力边界。同时说明文档库是用户个人上传的资料，区别于互联网搜索。

### 4.6 ask_user 连续调用限制

系统提示词中增加："同一轮对话中不要连续调用 ask_user 超过 2 次——如果用户已经做出了选择，立即基于选择结果行动或回复，不要追问过多。"

## 六、底层架构修复

### 6.1 `__LOCALID_` 线程分裂

**问题**：用户发送第一条消息时，transport headers 读取的 `remoteId` 为 `undefined`（`initialize()` 尚未完成），回退到 `__LOCALID_xxx` → 后端用这个 ID 创建了线程 → 第二条消息时 `remoteId` 已就绪 → 用了不同的 UUID → 消息被分裂到两个线程

**根因**：assistant-ui 的 `initialize()` 和 transport `send()` 并发执行，state 更新有延迟

**修复**：将 transport 的 `headers` 改为 `async`，轮询等待 `remoteId` 就绪（最多 500ms，10ms 间隔）

### 6.2 Unicode header 崩溃

`x-voice-summary` header 包含中文 → `Failed to read the 'headers' property from 'RequestInit': String contains non ISO-8859-1 code point` → 所有后续 fetch 请求全部失败 → 级联崩溃（历史加载失败、无法发消息）

修复：完全移除 header 方案，改用 `chat.setMessages()` 静默注入 + `POST /api/threads/:id/voice-messages` 持久化

### 6.3 `get_current_time` 工具移除

直接在系统提示词中注入当前时间（用户浏览器时区通过 `x-timezone` header 传递），省掉一次 LLM → 工具 → LLM 的 round-trip。

### 6.4 时区处理

Cloudflare Workers 的 `Date` 始终是 UTC。前端通过 `Intl.DateTimeFormat().resolvedOptions().timeZone` 获取浏览器时区，作为 `x-timezone` header 传给后端，后端用它格式化时间。
